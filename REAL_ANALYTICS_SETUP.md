# Real Website Views Analytics - Setup Guide

## Overview

You now have **real, tracked website analytics** instead of mock data. The system tracks page views, sessions, device types, and referrers automatically.

## What Was Added

### 1. Frontend Page Tracking (src/App.js)
- Automatic page view tracking on every route change
- Session ID generation and management (stored in sessionStorage)
- Device type detection (mobile, desktop, tablet)
- Referrer tracking
- Non-blocking tracking using `navigator.sendBeacon()`

### 2. Backend Tracking API (api/track-page.js)
- Receives page view events from frontend
- Validates and sanitizes input
- Stores data in Supabase `page_views` table
- Updates session records in `sessions` table
- Rate limiting (100 requests per session per hour)
- Graceful error handling

### 3. Database Tables (supabase/migrations/create_analytics_tables.sql)
Two new Supabase tables for storing analytics:

**page_views** - Every page view
- id, session_id, page_path, page_title, device_type, referrer, user_agent, created_at

**sessions** - Aggregated session data
- session_id, first_page_path, first_referrer, device_type, first_page_at, last_page_at, page_count

### 4. Admin Dashboard Updates (api/admin.js)
- `getWebsiteStats()` now queries real data from database
- Calculates: total views, unique visitors, bounce rate, session duration
- Aggregates: top pages, top referrers, device statistics
- Falls back to empty data if tables don't exist yet

## Data Structure

### page_views Table
```
id (auto)              - Unique view ID
session_id             - "session_1234567890_abc123def" (generated by frontend)
page_path              - "/pricing", "/signup", etc.
page_title             - "Pricing | FiiFit", etc.
device_type            - "mobile", "desktop", "tablet", or null
referrer               - URL referrer (null for direct traffic)
user_agent             - Browser user agent (for reference)
created_at (auto)      - Timestamp of view
```

### sessions Table
```
session_id (primary)   - Unique session identifier
user_id                - User ID if logged in (nullable)
first_page_path        - Page where session started
first_referrer         - Referrer for first page
device_type            - Device type (mobile/desktop/tablet)
first_page_at          - When session started
last_page_at           - Last page view time (updated on each view)
page_count             - Number of pages viewed in session
is_bounce              - Calculated if page_count == 1 after session ends
```

## Analytics Calculations

**Total Views** = Count of all rows in `page_views` table

**Unique Visitors** = Count of distinct `session_id` in `sessions` table

**Bounce Rate** = (Sessions with page_count == 1) / (Total sessions) × 100

**Avg Session Duration** = Average of (last_page_at - first_page_at) per session

**Top Pages** = GROUP BY page_path, COUNT(*), ORDER BY count DESC, LIMIT 5

**Top Referrers** = GROUP BY referrer, COUNT(*), ORDER BY count DESC, LIMIT 5

**Device Distribution** = GROUP BY device_type, COUNT(*), ORDER BY count DESC

## Performance

- **Tracking overhead**: ~1ms per page (non-blocking via sendBeacon)
- **Query overhead**: ~200-500ms to load admin dashboard (cached for 5 minutes)
- **Database size**: ~200 bytes per page view
- **Auto-cleanup**: Old data (>90 days) can be deleted via scheduled Vercel cron job

## Monitoring

### Check Data Collection

```sql
-- Last 10 page views
select page_path, device_type, referrer, created_at 
from page_views 
order by created_at desc 
limit 10;

-- Session summary
select 
  count(*) as total_sessions,
  sum(page_count) as total_pages,
  round(avg(page_count)::numeric, 2) as avg_pages_per_session,
  count(*) filter (where page_count = 1) as bounces
from sessions;

-- Device breakdown
select device_type, count(*) as sessions
from sessions
group by device_type
order by sessions desc;

-- Top pages
select page_path, count(*) as views
from page_views
group by page_path
order by views desc
limit 5;
```

## Troubleshooting

### No data appearing in admin dashboard

1. **Check tables exist**: In Supabase, go to Table Editor → verify `page_views` and `sessions` exist
2. **Check frontend tracking**: Open DevTools → Network → look for POST to `/api/track-page`
3. **Check API endpoint**: Verify `api/track-page.js` is deployed (run on Vercel)
4. **Check Supabase connection**: Run this in Supabase SQL Editor:
   ```sql
   select count(*) from page_views;
   ```

### "Unknown table" error in admin dashboard

- **Cause**: Tables not yet created in Supabase
- **Fix**: Run the SQL migration in Supabase SQL Editor (see Step 1 above)

### Tracking requests failing (404 or 500)

- **Check**: Is `/api/track-page.js` deployed? (verify in Vercel Deployments)
- **Check**: Are environment variables set? (SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY)
- **Note**: Even if tracking fails, your site keeps working (sendBeacon doesn't block)

### High bounce rate

- **Expected**: First-time visitors often bounce
- **Check**: Ensure only public pages are tracked (check skipPages in App.js)
- **Improve**: Add more engaging content to reduce bounces

### Data Retention
- Default: Keep all data (stored in Supabase)
- Optional: Delete data older than 90 days via cron job:

```javascript
// api/cron-cleanup-analytics.js
export default async function handler(req, res) {
  const ninetyDaysAgo = new Date(Date.now() - 90*24*60*60*1000).toISOString();
  
  const supabase = createClient(process.env.SUPABASE_URL, process.env.SUPABASE_SERVICE_ROLE_KEY);
  
  const { error: e1 } = await supabase
    .from('page_views')
    .delete()
    .lt('created_at', ninetyDaysAgo);
  
  const { error: e2 } = await supabase
    .from('sessions')
    .delete()
    .lt('first_page_at', ninetyDaysAgo);
  
  res.json({ deleted: !e1 && !e2 });
}
```
